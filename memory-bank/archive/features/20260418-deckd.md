# TASK ARCHIVE: deckd — initial delivery and PR-feedback rework

This document is the single canonical archive for the **deckd** Stream Deck daemon work on this repo: greenfield implementation (`deckd-initial`, Level 3), then targeted hardening from merged PR feedback (`deckd-rework-pr1`, Level 2), including a post-deploy hotfix for OnAir callback URLs. It supersedes the narrower file `memory-bank/archive/enhancements/20260418-deckd-rework-pr1.md`, which covered only the second phase.

---

## SUMMARY

**Phase A — Initial delivery.** Shipped a headless Python daemon around `streamdeck`: P2Pool toggle via systemd D-Bus with `systemctl` fallback, OnAir sign registration plus embedded Flask for GET/PUT state, `uv` + lockfile, tests with mocks, install artifacts (systemd, udev, polkit). Operator validated on target host (`kinglear`); tests reached 16/16. Roughly half of the eventual UX surfaced *after* the planned “complete” milestone: hardware showed units idling in `deactivating`, which drove deactivating blink PNGs, optional repeat-press `KillUnit` escalation, and polkit install docs—none of which were in the original plan.

**Phase B — PR-feedback rework.** Six accepted fixes: loopback Flask bind, blocking HTTP/subprocess off the asyncio loop and HID thread (`asyncio.to_thread`, `run_coroutine_threadsafe`), async `systemctl_*` fallbacks, `Manager.Subscribe()` at startup, done-callback logging on D-Bus handler tasks, tighter `.service` regex in tests. Nine new tests; suite at 25/25.

**Phase C — Hotfix.** Binding Flask to `127.0.0.1` without changing the URL registered with OnAir left pushes targeting the LAN IP while nothing listened there. Fixed by registering `http://127.0.0.1:{port}` and removing `deckd/netutil.py` (`guess_primary_ipv4`).

## PART I — Reflection notes inlined from `reflection-deckd-initial.md`

**Requirements vs outcome:** All six brief requirements met. Extra scope from real hardware: deactivating blink, kill escalation, polkit artifacts. Dropped: none; stale polkit comment referencing old paths/config was QA-fixed.

**Plan accuracy:** Component list and async/thread model matched implementation. Renames (`systemd_unit`, `deck_runtime`) and a `netutil` helper appeared; the latter was later deleted in the rework hotfix. The plan’s binary active/inactive model missed `deactivating` until hardware taught otherwise—worth enumerating all `ActiveState` values in future systemd-adjacent plans.

**Build / QA:** Planned build landed cleanly; a sequence of post-merge commits addressed real OnAir JSON, deactivating detection, blink UX, escalation, polkit. QA found only comment/doc drift after a file reorg—not logic bugs.

**Cross-phase analysis:** Refactor drift in comments/docs when moving `install/` layout; grep old paths across non-source files. Plan-to-reality gap on systemd middle states.

**Technical insights (initial):** Deactivating needs visible feedback + optional escalation; polkit unit filters are unreliable—user-scoped rules documented; model all six `ActiveState` values explicitly; `run_coroutine_threadsafe` works well with one clear main loop.

**Process insights (initial):** Creative phase correctly skipped for narrow open questions; hardware integration has a plan ceiling—budget post-deploy iteration; grep full tree on refactors; feature commits preserve narrative history.

## PART II — Reflection notes inlined from `reflection-deckd-rework-pr1` and rework archive

Surgical steps and tests passed each *named* change but missed **adjacent** coupling: loopback bind and the OnAir register callback URL must change together—one advertisement string, one listen address. Preflight validated local assumptions (`call_subscribe` naming, async callers) but not “what else still points at the old address?”

**Post-reflect hotfix:** Same host: deckd on `127.0.0.1:PORT`, OnAir elsewhere; LAN `curl` to `PORT` correctly refused; OnAir had been told to push to LAN IP via `guess_primary_ipv4()`—fixed to loopback.

**Million-dollar framing:** If thread-safety and non-blocking boundaries had been invariants from day one, less rework would have been needed; `systemPatterns.md` now documents the concurrency model.

**Process:** Add a grep/evidence step for bind/URL/port changes in the same diff; add at least one test that crosses an I/O boundary when that boundary moves, not only unit tests of changed functions.

## LESSONS LEARNED (combined)

- Initial: hardware reveals state machines and UX that diagrams miss; iterate after deploy.
- Rework + hotfix: bind address and any peer-visible callback URL are one logical change.
- Cross-thread asyncio tests should drain tasks before stopping the loop.
- `run_coroutine_threadsafe` returns `concurrent.futures.Future`; codebase uses duck-typed `Future` annotations.
